## Lens summaries

### 15/100 — Security: endpoint authentication, secret handling, SafePaperKite bypass, injection, netwo
Zero authentication on all REST and WebSocket endpoints makes every state-changing operation (ARM, KILL, position open/close, settings override) accessible to any network peer. Combined with CORS allow-all policy and no authorization checks, the system is defenseless against both direct network attacks and browser-based CSRF. Secrets at rest (KITE_API_SECRET, access_token) are world-readable in .env and access_token.json, not .gitignored, and error responses can leak credentials. The mandatory ARM gate is bypassed by manual_open endpoint.

### 22/100 — Quantitative validity of the edge: backtest realism, cost model, selection bias, spot-vs-o
The backtester deliberately trades the underlying (spot/futures) while the live engine buys single-leg options at ~0.5 delta with theta decay, IV crush and bid/ask spread up to 3% — none of these are modeled anywhere, so a spot-derived 'edge' is not evidence of a tradeable options edge. Cost model, fill model, and cache versioning are otherwise solid engineering, but there is zero walk-forward/out-of-sample discipline: any (instrument, interval, strategy) cell with as few as 1 trade can rank top and be promoted to the live universe, which is textbook multiple-comparisons cherry-picking across a universe x 6-timeframe x multi-strategy grid.

### 22/100 — Database integrity: transaction boundaries, constraints, migrations, SQLite concurrency
This system's database layer lacks foundational transaction-safety mechanisms critical for real-money trading. There are no rollback calls anywhere, no unique constraints enforcing position uniqueness, and no per-operation transaction boundaries. The broker's long-lived session accumulates uncommitted dirty state if any exception occurs mid-operation, which propagates silently into subsequent operations. API routes write directly to the shared broker session while competing with async engine loops, and a failed commit in either path leaves the capital ledger unreconciled. Together, these gaps create scenarios where cash and open positions can diverge from the true invariant, corrupting the ledger.

### 27/100 — Local book vs real broker account synchronization: reconciliation, phantom/ghost positions
The live broker has real, well-built ownership checks (can_bot_close, cancel-then-sell, orphan-confirm streaks) that prevent most naked/duplicate real orders in the common case. But the reconciliation layer cannot tell a genuinely flat account from a failed/throttled/token-expired Kite API call — both read as an empty positions list — so a sustained outage or the routine ~06:00 IST token expiry can make the engine mass-book real, still-open positions as 'externally closed' while doing nothing to protect them. Separately, the exact ghost-position incident class the system already suffered once (#17) was fixed only for the equity_intraday path, not for options — the platform's default and primary segment.

### 28/100 — Concurrency & async correctness: race conditions, lock coverage, double execution, event o
The risk loop was carefully hardened to offload blocking work off the event loop (asyncio.to_thread around mark_and_exit_positions), but the signal loop's entry path — the one place that actually places real orders — was never given the same treatment: scan_signals/process_entries run fully synchronously on the event loop, including live order placement with a ~30s blocking poll. That single omission undermines the two-lane design's core promise ('a slow poll never freezes the cockpit') and can suspend all SL/TP protection for every open position during any live entry. Several config-mutating REST routes are also plain sync handlers that mutate the runner's shared in-memory collections from a threadpool thread with zero synchronization against the asyncio.Lock the engine loops rely on.

### 30/100 — Crash/restart/failure recovery: durability, startup safety, mid-flight order handling, dou
The system has real, working safeguards for the "already-tracked position drifted" case (ownership guard, orphan reconcile with N-confirm hysteresis, exchange-side GTT/SL-M backstops), but crash recovery for the "order outcome unknown at the moment of death" case is largely absent: in-flight/pending-order state lives only in process memory and is wiped on every restart, and the options entry path (the platform's default, primary segment) never even attempts late-fill adoption. Combined with an unconditional startup "repair" heuristic that can silently inflate a genuinely partial live fill into a full lot, and no guard against two backend processes trading the same live account, a crash at the worst moment (mid-order) is the one failure mode this codebase is least prepared for.

### 32/100 — Order lifecycle & execution: state machine completeness, idempotency, retries, partial fil
The live order path has real, well-documented single-process safeguards (ownership guard, cancel-GTT-before-SELL ordering, partial-fill booking, an in-flight-order guard against double-sends within one run). But all of that safety state — _inflight, _pending_entries, order_fail_streak, last_entry_bar — lives only in Python instance dicts that are recreated empty on every process start, and there is no persisted order journal or exchange-side lookup by the pt-bot tag to recover in-flight orders after a crash. Two additional gaps are severe on their own: a filled order discovered late by _ensure_no_inflight is flagged but never adopted into the book for the options path, and a crash between a successful stop-cancel and the follow-up close order leaves a stale gtt_trigger_id that self-heal logic trusts blindly, wedging the position into a permanently 'looks protected but isn't' state.

### 32/100 — Infrastructure & operations: observability, loop survival, resource growth, deployment fra
This is a single-developer trading system running on a laptop with live real-money execution enabled. Critical infrastructure gaps exist: the two async loops catch exceptions but have no liveness monitoring, alerting, or watchdog; the health endpoint is a no-op; loop failures are silent (process stays alive but trading stops); there is no graceful shutdown protocol for in-flight orders; and the Vite dev server on localhost with caffeinate/no supervisor is extremely fragile. Database has grown to 1.4GB with no pruning. No structured observability exists beyond Telegram alerts tied only to trade events.

### 42/100 — Risk controls & financial safety rails: fail-closed enforcement, loss limits, kill switch,
The risk-control layer is unusually well-documented and mostly fail-closed by design (bounded runtime_config, per-tick single-threaded capital allocation, GTT-aware live close aborts), but two of its most important circuit breakers (unrealized-open-drawdown halt, re-entry cooldown) are shipped disabled by default and are confirmed still disabled in the actual running database, and the capital-vs-real-margin cross-check silently fails open exactly during the failure mode it exists to guard against (a Kite margins()/token failure). The kill switch also only squares off already-booked ledger positions, not orders that were in-flight/timed-out when it was pressed, so "KILL" is not a guaranteed instant-flat operation.

### 42/100 — Contract selection & market calendar: expiry, strikes, lot sizes, symbol mapping, market h
Expiry/strike resolution reads live from the Kite instrument dump (no hardcoded weekday, correctly handles NSE's shifted weekly-expiry days) and lot sizes are dump-driven with a sane MCX/NCDEX fallback -- good foundations. But the liquidity guard can be silently defeated by a zero-bid/zero-ask quote, there is no expiry-day cutoff so the picker will happily open a fresh position in an about-to-settle contract, and implied-vol/delta has no sanity bound so a stale or bad LTP can mispick the 'target-delta' contract with a nonsense IV.

### 48/100 — Cash/position/P&L accounting integrity: ledger invariant, charges accuracy, rounding, marg
The core paper-broker ledger math (options open/close/partial-close, charges, cash invariant) is careful and internally consistent, and equity MIS margin accounting is correctly segment-aware after a documented prior bug fix. However the invariant self-check is never run in production (only in an offline dry-run script), so drift already happened once live and would go undetected again; and the LiveBroker equity partial-close path is an acknowledged v1 gap that both mis-books P&L and leaves real shares unprotected. Charges are a reasonable approximation of Zerodha's schedule but are unverified against a live contract note.

### 58/100 — Strategy implementation & signal timing: lookahead bias, repainting, candle alignment, liv
Core signal math (EMA/z-score) and stale-signal/candle-freshness guards are careful and well-documented, with explicit fixes for known incidents (e.g. LODHA prior-session crossover). However two concrete defects survive: a clock inconsistency in the Kite historical-candle fetch that contradicts the codebase's own documented fix for the same class of bug, and a confirmed divergence between the validated backtest ratchet state machine and the actual live trailing-stop logic — meaning the trade-management the strategy was "proven" with is not the trade-management running with real money.

### 7. No guard against two backend processes trading the same live account
**Severity:** critical · **Likelihood:** ? · **Fix effort:** ?
**Files:** backend/app/main.py, backend/app/db/session.py — main.py:25-67 (lifespan creates its own EngineRunner + asyncio.Lock + tasks with no external coordination), db/session.py:20-25 (PRAGMA journal_mode=WAL / synchronous=NORMAL only — no advisory lock, no busy_timeout override, no pidfile)

EngineRunner's asyncio.Lock (runner.py:106) only serialises the two lanes *within one process*; nothing in main.py or db/session.py prevents a second uvicorn process (leftover from a previous session, an accidental double-launch, or a `--reload` misconfiguration) from opening the same SQLite file and independently running its own signal/risk loops and its own LiveBroker with a fresh, empty _inflight/_pending_entries.
